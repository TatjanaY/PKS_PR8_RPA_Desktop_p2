# Вариант 32 - **Кейс: «Автоматизация разрешения тикетов в системе поддержки»**

---

## 📖 Ситуация

После внедрения прототипа системы регистрации тикетов руководство компании **SupportFlow** приняло решение расширить её функциональные возможности.

В новой версии системы сотрудники должны иметь возможность:

* изменять статус тикетов (разрешать проблемы);
* назначать ответственных агентов поддержки;
* обрабатывать тикеты, поступающие из внешних источников данных.

Дополнительно планируется внедрение автоматизации обработки тикетов с использованием технологий **Robotic Process Automation (RPA)**. Роботы должны уметь:

* массово добавлять тикеты из файлов;
* изменять статусы тикетов;
* выполнять операции через пользовательский интерфейс приложения;
* обрабатывать системные окна и исключительные ситуации.

В рамках данной практической работы необходимо **доработать созданное на предыдущей практической работе приложение** и реализовать расширенный сценарий автоматизации.

---

# **Задание 1. Расширение функциональности приложения**

Необходимо **доработать созданное на предыдущей практической работе приложение**, добавив новые элементы управления и логику взаимодействия с таблицей тикетов.

---

## 🔘 Новые элементы управления

### ✔ **Resolve Ticket**

Изменяет статус выбранного тикета:

```json
{ "status": "Resolved" }
```

---

### 👤 **Assign Support Agent**

Назначает агента поддержки на выбранный тикет.

---

## ⚙️ Управление активностью элементов (IsEnabled)

Необходимо реализовать корректное управление доступностью кнопок:

* кнопки **Resolve Ticket** и **Assign Support Agent** должны быть **неактивны (`IsEnabled = false`)**, если строка таблицы не выбрана;
* при выборе строки кнопки становятся **активными (`IsEnabled = true`)**;
* после выполнения действия кнопки могут оставаться активными или сбрасываться при снятии выделения;
* кнопка **Resolve Ticket** должна быть неактивной, если статус тикета уже `"Resolved"`.

# **Задание 2. Разработка расширенного RPA-сценария**

Необходимо разработать процесс автоматизации в **Puzzle RPA**, выполняющий обработку тикетов.

---

## 🤖 Алгоритм работы робота

1. Запустить приложение.

2. Считать данные из Excel-файла.

3. Последовательно добавить **не менее 10 тикетов** в систему.

Для каждой записи:

* заполнить поля формы;
* проверить состояние кнопки **Add** (`IsEnabled`);
* при необходимости скорректировать ввод;
* добавить запись в таблицу.

4. Назначить агента поддержки для каждого тикета (используя кнопку **Assign Support Agent**).

5. Найти все тикеты с приоритетом **High**.

6. Для каждого такого тикета:

* выбрать строку в таблице;
* проверить, что кнопка **Resolve Ticket** активна (`IsEnabled = true`);
* изменить статус на `"Resolved"`.

# 🔎 Требования к корректности выполнения

Работа считается выполненной корректно, если:

### ✔ Интерфейс

* корректно реализовано управление `IsEnabled`;
* кнопки активируются только при выполнении условий;
* корректно обрабатывается выбор строки таблицы;

### ✔ Автоматизация

* робот корректно читает данные из Excel;
* добавляет все записи;
* находит и обрабатывает тикеты с приоритетом High;
* корректно работает с элементами интерфейса;

### ✔ Данные

* статусы обновлены корректно;
* данные не искажены;
