# Вариант 13 - **Кейс: «Автоматизация закрытия обращений клиентов»**

---

## 📖 Ситуация

После внедрения прототипа системы регистрации обращений руководство компании **SupportFlow** приняло решение расширить её функциональные возможности.

В новой версии системы сотрудники должны иметь возможность:

* изменять статус обращений (закрывать решенные);
* назначать ответственных операторов;
* обрабатывать обращения, поступающие из внешних источников данных.

Дополнительно планируется внедрение автоматизации обработки обращений с использованием технологий **Robotic Process Automation (RPA)**. Роботы должны уметь:

* массово добавлять обращения из файлов;
* изменять статусы обращений;
* выполнять операции через пользовательский интерфейс приложения;
* обрабатывать системные окна и исключительные ситуации.

В рамках данной практической работы необходимо **доработать созданное на предыдущей практической работе приложение** и реализовать расширенный сценарий автоматизации.

---

# **Задание 1. Расширение функциональности приложения**

Необходимо **доработать созданное на предыдущей практической работе приложение**, добавив новые элементы управления и логику взаимодействия с таблицей обращений.

---

## 🔘 Новые элементы управления

### ✔ **Close Request**

Изменяет статус выбранного обращения:

```json
{ "status": "Closed" }
```

---

### 👤 **Assign Operator**

Назначает оператора на выбранное обращение.

---

## ⚙️ Управление активностью элементов (IsEnabled)

Необходимо реализовать корректное управление доступностью кнопок:

* кнопки **Close Request** и **Assign Operator** должны быть **неактивны (`IsEnabled = false`)**, если строка таблицы не выбрана;
* при выборе строки кнопки становятся **активными (`IsEnabled = true`)**;
* после выполнения действия кнопки могут оставаться активными или сбрасываться при снятии выделения;
* кнопка **Close Request** должна быть неактивной, если статус обращения уже `"Closed"`.

# **Задание 2. Разработка расширенного RPA-сценария**

Необходимо разработать процесс автоматизации в **Puzzle RPA**, выполняющий обработку обращений.

---

## 🤖 Алгоритм работы робота

1. Запустить приложение.

2. Считать данные из Excel-файла.

3. Последовательно добавить **не менее 10 обращений** в систему.

Для каждой записи:

* заполнить поля формы;
* проверить состояние кнопки **Add** (`IsEnabled`);
* при необходимости скорректировать ввод;
* добавить запись в таблицу.

4. Назначить оператора для каждого обращения (используя кнопку **Assign Operator**).

5. Найти все обращения с приоритетом **High**.

6. Для каждого такого обращения:

* выбрать строку в таблице;
* проверить, что кнопка **Close Request** активна (`IsEnabled = true`);
* изменить статус на `"Closed"`.

# 🔎 Требования к корректности выполнения

Работа считается выполненной корректно, если:

### ✔ Интерфейс

* корректно реализовано управление `IsEnabled`;
* кнопки активируются только при выполнении условий;
* корректно обрабатывается выбор строки таблицы;

### ✔ Автоматизация

* робот корректно читает данные из Excel;
* добавляет все записи;
* находит и обрабатывает обращения с приоритетом High;
* корректно работает с элементами интерфейса;

### ✔ Данные

* статусы обновлены корректно;
* данные не искажены;
